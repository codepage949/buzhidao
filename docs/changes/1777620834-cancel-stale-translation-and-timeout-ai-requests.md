# 오래된 번역 요청 취소 및 AI 요청 timeout 추가

## 배경

번역 요청이 초반에는 빠르다가 한 번 느려진 이후 계속 느리게 느껴지는 문제가 있다.
현재 팝업은 최신 request id만 반영하지만, Rust 백엔드에서 이전 AI 요청 자체는 취소하지 않는다.
느린 요청이 남아 있으면 이후 요청과 네트워크/게이트웨이 자원을 공유하며 지연이 누적될 수 있다.

## 계획

1. 번역 요청을 별도 async 작업으로 실행하고, 새 번역 요청이 시작되면 이전 작업을 abort한다.
2. 팝업 또는 오버레이를 닫을 때도 진행 중인 번역 작업을 abort한다.
3. AI Gateway용 `reqwest::Client`를 명시적인 timeout/idle 연결 정책으로 생성한다.
4. AI 응답은 HTTP 상태 코드를 먼저 확인해 429/5xx 등 오류를 명확히 표시한다.
5. 핵심 로직은 단위 테스트로 고정한다.

## 구현 내용

- `TranslationTaskState`를 추가해 현재 번역 작업의 abort handle을 저장한다.
- `select_text`는 팝업 표시 후 즉시 반환하고, 번역 결과 emit은 별도 작업에서 수행한다.
- 새 번역 요청이 시작되면 이전 번역 작업을 abort한다.
- 팝업 또는 오버레이를 닫을 때 진행 중인 번역 작업을 abort한다.
- AI 요청에 connect timeout, 전체 request timeout, idle connection timeout을 설정한다.
- HTTP 오류 응답은 본문 일부와 상태 코드를 포함해 사용자에게 전달한다.
- AI 요청 완료 시 소요 시간을 로그로 남긴다.

## 검증

- 이전 번역 작업 교체 시 이전 작업이 취소되는지 테스트했다.
- HTTP 오류 상태 메시지 생성과 빈 응답 처리 테스트를 추가했다.
- `cargo test` 통과.
