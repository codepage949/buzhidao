# 멀티플랫폼 대비 플랫폼 추상화 준비

## 배경

- 현재 앱은 Windows에서 먼저 동작을 맞춘 상태지만, 코드상 전역 입력 감지, 화면 캡처, 오버레이 창 제어가 한 곳에 섞여 있다.
- 이 구조는 macOS/Linux 대응 시 운영체제별 분기 지점을 찾기 어렵게 만든다.
- 지금 단계에서 멀티플랫폼 지원을 완성하기보다, 운영체제 민감 코드를 모듈 경계 뒤로 옮겨 이식 비용을 낮추는 준비가 필요하다.

## 목표

- 전역 입력 감지, 화면 캡처, 오버레이 창 제어를 플랫폼 모듈로 분리한다.
- 현재 Windows 동작은 유지하면서 상위 호출부가 플랫폼 구현 세부사항에 직접 의존하지 않게 한다.
- 멀티플랫폼 확장 시 어디를 교체해야 하는지 코드 구조상 분명하게 만든다.

## 구현 계획

1. 플랫폼 추상화 대상 기능과 현재 호출 지점을 정리한다.
2. `platform` 모듈을 추가하고 전역 입력, 캡처, 오버레이 표시 책임을 옮긴다.
3. 앱 진입점은 플랫폼 모듈의 진입 함수만 호출하도록 정리한다.
4. 테스트와 회귀 검증으로 기존 동작이 깨지지 않았는지 확인한다.

## 구현 내용

- `src/platform.rs`를 추가해 전역 캡처 단축키 등록, 활성 화면 캡처, 오버레이 준비 동작을 모았다.
- `src/lib.rs`는 더 이상 `rdev`와 캡처/오버레이 세부 구현을 직접 다루지 않고 `platform` 모듈 함수를 호출하도록 정리했다.
- 전역 입력 감지는 `install_capture_shortcut`, 화면 캡처는 `capture_active_screen`, 오버레이 표시는 `prepare_overlay_for_capture` 경계 뒤로 숨겼다.
- 현재 구현은 여전히 기존 Windows 중심 동작을 유지하지만, 이후 macOS/Linux 분기 지점이 `platform` 모듈에 모이도록 구조를 바꿨다.
- `PrintScreen` 감지 규칙은 `platform` 모듈 테스트로 검증할 수 있게 분리했다.

## 리팩토링 검토

- 전역 입력 등록, 캡처, 오버레이 표시가 `lib.rs`에서 분리되어 중복과 결합도가 줄었다.
- 다음 단계의 실제 멀티플랫폼 대응은 `platform` 모듈 안에서 OS별 구현을 나누는 형태로 진행하면 된다.
