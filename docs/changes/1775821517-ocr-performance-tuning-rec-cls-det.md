# OCR 성능 3차: rec·cls·det 경로 튜닝

## 배경

최근 측정값:

- det 전체: 2169ms
- det 타일 3x3: 4934ms
- cls (212 박스): 2017ms
- rec (212 박스): 15643ms

det 타일은 4x4에서 3x3으로 줄이며 크게 개선됐지만,
전체 시간의 대부분은 여전히 `cls + rec`가 차지한다.

## 분석

### 1. rec는 아직 박스당 비용이 높다

`15643ms / 212 박스 ≈ 74ms/박스`로 여전히 느리다.

현재 구조는 이미:

- 너비 정렬
- 청크 배치
- GPU 배치 실행

을 적용했지만, `MAX_W=960` 상한과 `REC_BATCH_SIZE=16`은
화면 OCR 기준으로 아직 보수적이다.

### 2. cls는 대부분 화면에서 낭비일 가능성이 높다

일반 UI/문서 캡처에서는 180도 뒤집힌 텍스트가 거의 없다.
그런데 현재는 모든 박스에 대해 항상 cls를 수행해 약 2초를 소비한다.

## 목표

1. `rec`의 time step 상한을 추가로 낮춘다.
2. `cls`는 전체 박스 전수 검사 대신 샘플 검사 후 필요할 때만 전체 실행한다.
3. 런타임 로그를 남겨 실제 병목이 어디인지 계속 확인 가능하게 유지한다.

## 변경 계획

### `src/ocr/rec.rs`

- `MAX_W`: `960 -> 768`
- 워밍업 너비 목록도 새 폭 상한에 맞춰 조정
- `MAX_W` 상한 테스트 추가
- 전 구간 가변 청크는 오히려 느려져 적용 보류
- 대신 큰 폭 tail 구간만 더 작은 배치로 처리하는 하이브리드 청크 적용
- 측정 결과 320~640 구간도 병목이라 중대형 구간 배치를 추가로 축소
- 측정 결과 236~486 구간도 병목이라 중간 폭 구간 배치를 추가로 축소
- 측정 결과 500+ 폭 tail도 여전히 비싸서 tail 배치를 한 단계 더 축소

### `src/ocr/mod.rs`

- cls 전수 실행 전 샘플 배치 검사 추가
- 샘플에서 회전 박스가 없으면 나머지 박스는 cls를 생략하고 정방향으로 처리
- cls/rec 직전 초소형 박스를 보수적으로 필터링
- 샘플 cls는 단일 오탐으로 전체 cls를 켜지 않도록 임계치 적용
- 화면 OCR 기준으로 극단적인 세로형 박스는 OCR 전 제외
- 측정 결과 샘플 회전 2개도 전체 cls 비용 대비 이득이 작아 임계치를 추가 상향
- `OCR_ENABLE_CLS` 환경변수로 cls 자체를 옵션화하고 기본값은 `false`
- 샘플/전체 실행 여부와 필터링 개수를 로그에 남김

### `src/services.rs`

- tile/full 병합 후 전체 박스에 대해 한 번 더 NMS 적용
- full 박스 보충 기준을 더 공격적으로 조정해 tile이 이미 잡은 영역 재추론을 줄임
- NMS 억제 IoU/포함 비율 기준도 추가로 강화해 rec 입력 수를 더 줄임
- 같은 줄의 거의 동일한 높이/세로 위치 중복 박스를 잡는 수평 텍스트 전용 억제 조건 추가
- dense `3x3` 타일 모드에서는 full det를 생략하고 tile-only 경로 사용

## 기대 효과

- `rec`: 긴 박스가 많은 화면에서 LSTM time step 감소
- `rec`: 초소형 후보 제거로 박스 수 자체 감소 기대
- `cls`: 회전 텍스트가 없는 일반 화면에서 약 2초 절감 기대
- `det->rec`: tile/full 중복 박스 감소로 rec 입력 수 감소 기대
- 전체: 현재 24초대에서 추가 하향 기대

## 검증 계획

- 단위 테스트: `target_width` 상한, cls 샘플 인덱스 선택, 초소형 박스 필터링
- 실행 로그 확인:
  - `rec 너비 분포`
  - `rec 청크`
  - `cls 샘플`
  - `ocr 박스 필터`
  - `rec (N 박스, rotate180 X)`

## 최종 판단

- `OCR_ENABLE_CLS=false` 기본값은 유지할 가치가 있다.
- `dense 3x3`에서 `full det`를 완전히 생략하는 tile-only 경로는 속도는 크게 개선했지만
  실제 인식률 저하가 확인되어 기본 경로로 채택하지 않는 편이 낫다.
- 따라서 이후 작업은
  - 성능 유지 작업
  - 인식률 향상 작업
  를 분리해 진행하는 것이 적절하다.
