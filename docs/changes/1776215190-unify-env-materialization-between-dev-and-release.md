# 개발/배포 .env 물질화 동작 통일

## 목적

- 개발 모드와 배포 모드에서 `.env.example` 기반 설정 초기화 동작을 동일하게 맞춘다.
- 기존 `.env`에 남아 있는 깨진 줄이나 예전 설정 찌꺼기 때문에 앱 시작이 실패하지 않게 한다.

## 구현 계획

1. 개발/배포 모드 모두 동일한 `.env` materialize 경로를 사용한다.
2. `.env.example`에 정의된 키만 유지하고, 기존 `.env`의 malformed line은 무시한다.
3. 변경 후 `cargo test`로 설정 로드와 머지 동작이 유지되는지 확인한다.

## 구현 내용

### 설정 로드 경로 통일

- 앱 시작 시 개발 모드도 배포 모드와 동일하게 `.env.example`을 기준으로 `.env`를 materialize한다.
- 이후 두 모드 모두 `Config::from_env_file(...)`로 같은 로드 경로를 사용한다.
- 이 변경으로 개발 모드만 별도로 `dotenv()`를 타던 경로를 제거했다.

### `.env` 정리 방식 변경

- 기존 `.env`에서 정상적인 `KEY=VALUE` 형태로 읽히는 값만 사용한다.
- `.env.example`에 정의된 키에 대해서만 기존 값을 override한다.
- `.env.example`에 없는 키와 형식이 깨진 줄은 새 `.env`에 반영하지 않는다.
- 따라서 과거 `SYSTEM_PROMPT` 멀티라인 값 같은 잔여 데이터가 있어도 앱 시작 전에 자동 정리된다.

## 테스트

- `cargo test --manifest-path app/Cargo.toml`

## 리팩토링 검토

- 개발/배포 분기에서 서로 다른 설정 로드 함수를 사용하던 중복을 제거했다.
- `Config::from_env(...)`는 더 이상 사용되지 않아 삭제했다.
